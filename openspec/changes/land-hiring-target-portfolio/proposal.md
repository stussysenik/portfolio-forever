# Proposal: Land Hiring-Target Portfolio

## Why

The portfolio now needs to act less like an archive and more like a hiring instrument.

The target role set is no longer vague. The active company list and job pages cluster around:

- design engineering
- AI product / experience design
- frontend systems engineering
- multi-platform product craft
- trust-heavy interfaces for expert users

The current repo already contains the ingredients, but the homepage still spreads proof too evenly.
It needs a clearer hiring thesis, a stronger recruiter read in the first minute, and a permanent
record in `openspec` of what the portfolio is trying to win.

## What Changes

- Record the target hiring clusters and source job research in one active spec.
- Add a permanent code-level hiring target model for the public experience.
- Reframe the homepage hero as explicit hiring proof, not only identity + showcase.
- Prioritize proof routes that answer recruiter questions quickly: `works`, `cv`, `process`.

## Success Criteria

- [ ] The repo contains an active `openspec` change that states the hiring target explicitly.
- [ ] The homepage above the fold communicates the target role fit in under a minute.
- [ ] The public site makes clear that this is proof of shipped competence, not only taste.
- [ ] The first-pass hiring framing is compatible with later admin/content tuning.

## Non-Goals

- Turning the public site into a literal job-application microsite for one company
- Hard-coding recruiter-specific copy into every section
- Solving every CV/content extraction problem in the same batch
